El Problema y la Motivacion

Limitaciones Actuales

  • La gestion de aulas informaticas es hoy una tarea lenta y manual.
  • El software comercial tradicional se limita a la supervision visual de pantallas.
  • No existe capacidad de intervencion en la infraestructura fisica de red.

La Necesidad Real

  • Es necesario aislar el trafico actuando directamente sobre el puerto del switch.
  • Integrar la monitorizacion de software con el hardware MikroTik del centro.
  • Proveer una solucion centralizada que funcione estrictamente en tiempo real.

Objetivos del Proyecto

Centralizacion

Interfaz web unica para gestionar multiples aulas y dispositivos de red de forma remota.

Tiempo Real

Control inmediato de pantallas, hardware y latidos de conexion mediante WebSockets.

Network Fencing

Aislamiento fisico de red para permitir solo el acceso a recursos autorizados dinamicamente.

Stack Tecnologico I: Logica y Conectividad

Python 3.13 + Django 5

Elegido por su arquitectura modular y el potente ORM, que permite gestionar modelos complejos de red con alta seguridad nativa.

librouteros

Libreria especializada en la comunicacion binaria nativa con la API de MikroTik, permitiendo ejecuciones de red en milisegundos.

Fernet (AES-128)

Implementacion de criptografia simetrica para cifrar las contraseñas de los equipos de red antes de su persistencia en la base de datos.

Stack Tecnologico II: Tiempo Real y Agente

WebSockets + Daphne

Uso de servidor ASGI para canales de datos persistentes. Imprescindible para recibir capturas de pantalla y enviar comandos en vivo.

Redis 7

Actua como Channel Layer para Daphne y gestiona el sistema de latidos (heartbeat) mediante notificaciones keyspace por TTL.

mss (Python)

Motor de captura de pantalla ultrarrapido integrado en el agente. Permite monitorizacion visual con impacto nulo en la CPU.

Stack Tecnologico III: Interfaz y UX

React 19 + Vite

Construccion modular basada en componentes reactivos. Vite garantiza tiempos de carga y desarrollo instantaneos.

Material UI 7

Biblioteca de diseño que aporta una estetica profesional y componentes accesibles que agilizan el uso del dashboard docente.

Nanostores

Gestion de estado atomica. Se utiliza para cachear imagenes enviadas por sockets de forma eficiente en el navegador.

El Corazon: Network Fencing

Estrategia Dual de Bloqueo

El sistema protege el entorno mediante una doble capa de aislamiento fisico automatizado.

  • Capa L2 (Enlace): Modificacion dinamica del split-horizon en el bridge para aislar ordenadores entre si.
  • Capa L3 (Red): Inyeccion de reglas Firewall Filter que bloquean internet pero permiten recursos educativos.

Aislamiento Real

Operacion ejecutada en hardware MikroTik.

Monitorizacion y Heartbeat

Deteccion Inmediata

  • El agente emite un latido (heartbeat) vital cada 60 segundos.
  • El servidor renueva una clave en Redis con TTL de 2 minutos.
  • Si la clave expira, Redis dispara una notificacion asincrona.
  • El dashboard marca el equipo como offline al instante.

Telemetria Visual

  • Capturas de pantalla codificadas en Base64.
  • Compresion en origen para no saturar el ancho de banda.
  • Reporte continuo de usuario local y direccion IP activa.

Complicaciones Encontradas I: Sistemas

Wayland vs X11

En Linux modernos, Wayland impide por seguridad que aplicaciones capturen pantalla de otros procesos. Se exigio forzar sesiones X11 en el aula.

Servicios Windows

El registro del agente como servicio puro (NSSM) era inestable. Solucionado usando el Task Scheduler con privilegios de sistema vinculados al arranque.

Complicaciones Encontradas II: Red

Certificados WSS

Los navegadores rechazan WebSockets seguros en red local sin certificados validos. Resuelto configurando Caddy como Autoridad Certificadora (CA) local.

Puertos del Switch

Dificultad tecnica para asociar IPs con sus puertos fisicos. Requirio procesar iterativamente las tablas ARP y CAM (MAC) de MikroTik para el autodescubrimiento.
